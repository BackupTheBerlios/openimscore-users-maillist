Hi Roman, 

As Franz said, you could define an IFC so that it triggers a notification to your AS when the user registers in the core. Then you should attach this IFC to the users' Service Profile (to the SPs of all the 1 000 000 users). And to avoid massive notifications to your AS, you could switch that IFC OFF/ON and viceversa whenever it is necessary by means of the DSAI: You should create a DSAI, attach the IFC (the one previously defined) to the DSAI and configure the DSAI value for each user. The default value for all the users should be INACTIVE, so that every user registering does not trigger a notification to the AS. And when you need that trigger activated for a certain user, you could switch the DSAI value to ACTIVE for that particular user.

Going on with your example, you could switch the DSAI for the destination host (user B) to ACTIVE when you detect that he is not registered. When user B registers, the AS will be notified about that by the S-CSCF and will be able to resend the stored message to user B.

The DSAI is already implemented (please refer to the branch "07_Dynamic_Service_Activation" in the FHoSS) and I hope it is merged soon into the trunk. Maybe Alberto could tell us when he expects merging that ;)
