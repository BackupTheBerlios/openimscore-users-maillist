Hi Tomasz,
 
> This is strange because acording to RFC 3261
> The To and From fields in the CANCEL request must be identical to those
> in the request being cancelled, including tags.
> Here cancelled request (INVITE) had no tag in To header.

Yes, you are right. The To-tag is not necessary.

Now it is hard to guess because the important INVITE request is missing in
your trace due to packet fragmentation.
 
> Another thing that bothers me is that both pcscf and scscf send CANCEL
> request which contains more then one Via header.
> The same as before accoreding to rfc 3261 it
> should have a single Via header with value matching the
> top Via value in the request being cancelled.

Yes, you are also right, but it obviously does not hurt. You should
eventually put the question on the SER mailing list. I assume that anyone
has already detected the issue and there is an answer.

The similar situation which I detected earlier is also with the 100 Trying
response. A 100 Trying response is also handled hop-by-hop (like CANCEL
request) and therefore only needs one Via header field.
