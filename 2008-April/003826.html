Hi all,
 
Using the Open IMS Core I have come across an issue, namely using an IMS B2B user agent that is not using the service route it has received in the 200OK from the REGISTER, when sending an INVITE SIP header Route goes to the pcscf  (example: Route: <sip:192.131.160.4:4060;lr> instead of this received service route in the 200 OK: Service-Route: <sip:orig at scscf.openims.core:6060;lr> ) 
 
At first a 400 message was return "Bad request - Not following indicated service routes"
 
Looking at the configuration file ( pcscf.cfg) (This is a snippet of the config:)
 
if (!P_follows_service_routes()){    #Variant 1 - deny access to the network  sl_send_reply("400","Bad Request - Not following indicated service routes");  break;  #Variant 2 - enforce routes and let the dialog continue     #P_enforce_service_routes();
 
I enabled P_enforce_service_routes(); so I don't receive any longer a 400 back but yet there is no answer from Open IMS Core. Is there any work around this? Going through the RFC, it says that the Service-Route is not compulsory to be followed - is this specific for Open Ims Core only?
