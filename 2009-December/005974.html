Thanks Dragos for the information


For your second point on storage of reginfo document, I see in the code
(r_notification_process), we are calling update_r_contact with service
route related args sets to NULL and in that function we simply return if
this is NULL. So effectively there is no update happening on any
database maintained by PCSCF for such Notifications. 
And this is required as per RFC 3680 because a subscriber maintains
tables as per subscriptions and update them as per Partial or Full data
coming in Notify, which is the basic 3680 processing of the reginfo doc.
In other words, P-CSCF does not comply 3680 in this respect as well as
of today. It just parses the xml.

Let me know if I am missing something here.

Thanks
Preeti

Hi Preeti,

regarding the version, you are right - this feature is not yet 
implemented. We're trying to cover as much functionality with as little 
effort. While the version would be helpful, this is only true on some 
limit situations. Anyway, patches and improvements are welcomed.

Regarding the storage of the reginfo document, the P-CSCF parses the 
information and keeps it's local version of the registrar synchronized 
with the status on the S-CSCFs. I don't see any need in keeping the 
entire document in memory after this update operation. Or is there any?

Cheers,
-Dragos
