Hi Luca,

> I'd like to re-route NOTIFY messages between two different
> presence services using IMS core components, i.e., S-CSCF.

If I understand you right you want to use IMS and IFCs to route SIP
requests. This is a wrong view on the architecture of IMS and cannot be
done. You should use a SIP proxy for routing of SIP requests.

Or maybe you have something in mind I do not understand. In this case you
should elaborate more what you are doing.

> Hence, after some time spent on modifying default trigger point
> to achieve my goal, I had a look to iFC FAQ, and found this:
> "The Initial Filter Criteria can only match Initial Requests, like
> INVITE, REGISTER, SUBSCRIBE, MESSAGE, etc.
> Subsequent messages (like BYE, NOTIFY, INVITE inside a
> dialog, etc) are not checked or forwarded."
> 
> Hence, I was wondering if it was possible to modify iFC behavior
> so to re-route "subsequent" messages (in my case NOTIFY message).
> In the case, what should I dig into?

If you want to route a NOTIFY to an AS you have to trigger on the dialog
initiating request (which is SUBSCRIBE in case of NOTIFY) and set a
Record-Route header field.
