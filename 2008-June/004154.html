> I attach another capture with the same problem (RAR file).

The trace-file now is perfect. I have created a scenario (attached) which
makes the analysis much easier.

> In Asterisk I defined the users jane and john, but Asterisk is not
> recognizing them.

As you can see Asterisk tries to forward the INVITE (packet 23) but the
packet is rejected by S-CSCF because it does not follow IMS rules. The
INVITE does not contain the 2nd route header (packet 21) which enables the
S-CSCF to recognize the packet as a continuation of a previously forwarded
INVITE to an AS. Therefore the S-CSCF assumes that the INVITE is a
terminating one arriving from somewhere. Now it looks on the request URI to
find the user, but the URI offered by Asterisk does not have a user part
(packet 23). So it cannot handle the request. 

> Anyway the call is made, but does not end.

I assume that the AS is provisioned as with the Default Handling "Session -
Continued". The S-CSCF therefore continues the session setup after timeout
(but I cannot verify a reasonable timeout, there are only 240 ms between
packet 19 and 47) without involving Asterisk anymore.

> The S-CSCF returns a Status 403 Forbidden when the UE sends a Request BYE.

I cannot explain quickly why the BYE is rejected. An S-CSCF log could help
to identify that problem.

A quick summary: The Asterisk obviously is not well suited as an AS for IMS.
