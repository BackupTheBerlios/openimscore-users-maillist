Hello Jean-Yves,

> #2 after some time, IMS client wants to register IMPU2. My understanding
> is that it shall build a new REGISTER request for IMPU2
> and send it using IPSec tunnel set up in #1.

According to TS 24.229 §5.1.1.4.1 the UE has two choices:

The UE shall perform the registration of additional public user identities
either:
- over the existing set of security associations or TLS sessions, if
appropriate to the security mechanism in use, that is associated with the
related contact address; or
- via an initial registration as specified in subclause 5.1.1.2.

> In this case, whatever the authorization field in this 2nd register, the
> S-CSCF shall not trigg AKA because S-CSCF can validate that
> the IMPI associated to IMPU2 is already authenticated (done with IMPU1
> registration).
> afterwards IMS client will be able to re-register / de-register IMPU2
> independantly from IMPU1

Yes, if it uses the first option (re-use of an existing set of security
associations). This leads to an integrity-protectde="yes" parameter in the
Authorization header and therefore no MAR/MAA sequence is triggered.

> am I right? is it the way OSIMS works? (not able to test it for the
> moment)

Yes, you are right and so I expect that OpenIMSCore will work (not tested
yet).
