Dear Dragos,

please note that I did not want to propose to remove anything from the website, but that the rationale behind my questions was just to understand how the OpenIMScore relates with the ONLY standard officially existing (at least for 3GPP, 3GPP2, TISPAN and CableLabs) nowadays on IMS, that is 3GPP.



To me, particularly in an environment like IMS, where multiple parties seem to be interested in, the compliance to the standard seems to be the only way forward, however this is only my personal view. I am not representing the 3GPP standard, of course: I just attend 3GPP and this is why I know how difficult has been the Common IMS process, that implies the satisfation of several competing needs on a sistem (IMS) that suddenly, after years on "independent life" was supposed to become COMMON... It has been very difficul also because TS 33.203 was originally written having 3GPP-only needs and requirements in mind, that means, for instance, users with a UICC, whereas TISPAN (just to give an example) is reluctant to adopt the UICC (that is the "smartcard").

The Common IMS process (started within the 3GPP Rel-8) was decided to accommodate the IMS needs coming from various players and as it has be a real nightmare to specify, now I hope it will be helpful for the implementors...  ;o))



If IMS authentication mechanisms other than those considered by TS 33.203 Rel-8 are left in the OpenIMSCore website it could be fine, I guess, however I cannot figure out how they will be handled IN PRACTICE by the "coexistence mechanism" described within the Annex P of TS 33.203 Rel-8, provided that the coexistance mechanism will be implemented in the future by the OpenIMCcore website.

In my view the standard can be used as a guideline as long as you comply with it, but if you start deviating (e.g. adding options that might be absolutely reasonable but that are not foreseen by the standard), then there is the chance that you cannot rely anymore on the conclusions/guidelines written in the standard (in my view, as a minimum you have to doublecheck everything, as you never know, that means wasting all the efforts done to write the standard...).



However, to sum up the conclusions of our discussion:

1.    OpenIMSCore implements a set of possible IMS authentication mechanism that goes beyond the list considered by 3GPP TS 33.203. However the IMS authentication mechanisms considered by TS 33.203 Rel-8 should be implemented by OpenIMSCore. More in detail:

a.    The mechanism called "SIP Digest (3GPP)" in the OpenIMSCore is supposed to be compliant with the "SIP digest" considered by TS 33.203.

b.    The mechanism called "NASS Bundled (ETSI)" in the OpenIMSCore should be the implementation of the "NASS Bundled" considered by TS 33.203.

c.    The mechanism called "Early-IMS (3GPP)" in the OpenIMSCore should be the implementation of the "Early IMS" (now renamed "GIBA") considered by TS 33.203.

d.    The IMS AKA mechanism specified by 3GPP since its Rel-5 IS IMPLEMENTED by the OpenIMSCore as "3GPP Digest-AKAv1-MD5". However, could you kindly explain me why the HSS user interface does not show it beside the other implemented IMS authentication mechanisms?

2.    The "coexistence" mechanism described within the Annex P of TS 33.203 Rel-8 is NOT implemented yet by the OpenIMSCore.



Can you agree on the above conclusions?

It has been a quite long discussion, however it has been very useful, at least for me.

Many Thanks again!



Best Regards.



Mauro

3GPP Digest-AKAv1-MD5, Digest-AKAv1-MD5 and FOKUS Digest-MD5 are 3

different authentication methods, which share a lot of common code. We

implement all 3. The first one is what 3GPP had from the beginning, AKA,

while the last is the simplest solution to having MD5 authentication.

This and the P-CSCF capability of enforcing Service-Routes even if the

client did not supports it, makes it possible to use virtually any SIP

client with IMS. And I think this was one of the big positive steps that

we took forward.



Then we called it FOKUS as to not confuse people on where does it come

from or what to expect on the Cx interface. The functionality is sooo

simple in fact that I wonder how 3GPP missed the opportunity. So

Digest-AKAv1-MD5 is a mechanism through which the secret key is piped

through AKA to get a password for MD5, plus additional keying material

for other purposes. If you input directly the secret key as a password

for MD5, you get the classical Digest-MD5, which works with every SIP

client. Sure, you can't use IPsec any more on Gm because no other keys

are generated, it is inherently less secure and, even worse, the

password is transmitted in clear on the Cx interface (although this

should be IPsec protected), but it just made sense and it is also simple.



 From a UE perspective all these Digest-MD5 algorithms (FOKUS, ETSI,

CableLabs, 3GPP) are exactly the same and we just needed one to get past

the client issue. If we enforce all the stuff with "MUST" in the

standard, but none that make sense in reality, then think how hard it

would be to test some simple Presence Server let's say if you need: an

AKA client, a USIM/ISIM ready for it, a client supporting IPsec, a full

Diameter stack and so on. IMS is slow to get off the ground as it is, so

we're just trying to accelerate adoption.



Although I don't think that this is what you meant, if you're looking at

it as we should not call it Open IMS Core as IMS would be a kind of

trademark from 3GPP, which requires full compliance or nothing, we're

out of scope as we don't care about this kind in our R&D targets and

we'd probably rather rename the project. We're not claiming any

compliance to any release, but just consider that we implemented the

core features from many of them.

I agree. We actually wanted this Common IMS so much as it is not funny

for this to happen (years) and we don't think that we can afford it. If

IMS takes too long to get adopted, why bother and not just go directly

to the Next-NGN? I don't want to go now into some long discussion about

project, we should accept any potential improvements which are in scope,

regardless if they come from an approved source or not. We're trying to

achieve some meaningful merge between the Internet and the telecom, so I

think that we have to adapt the telecom ways of standards and components

Then we'd, of course, like to have our project compliant to the

standards, especially as it's a great independent tool for testing and

And it is a pity, especially as we already evaluated this coexistence,
