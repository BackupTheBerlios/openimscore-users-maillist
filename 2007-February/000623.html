>> Sudhakar,
>>
>> Babu's suggestion is exactly what I'm currently
>> using, and it works fine
>> with various clients (not all of them though),
>> including X-Lite on
>> Windows. This is possible because recent svn
>> snapshots support MD5 both
>> in the CSCFs and in the HSS. Also, even if you can't
>> select the
>> algorithm in X-Lite, it does support MD5 when
>> responding to an
>> authorization challenge.
>>
>> Here is what I think needs to be modified:
>> * In the HSS (e.g. using the web i/f), change the
>> Authorization method
>> and algo to Digest-MD5 and MD5 in the Private
>> Identity profile of the
>> user.
>> * In scscf.cfg, change the
>> registration_default_algorithm parameter to
>> MD5. This is required because otherwise the S-CSCF
>> and HSS would produce
>> an AKAv1-MD5 challenge to which the X-Lite wouldn't
>> know how to respond
>> and would "therefore" not respond at all. Other
>> clients show different
>> behaviors: some would provide an MD5 response but
>> without changing the
>> algorithm field, therefore causing authorization
>> failure; some would do
>> the same but change the algorithm field and these
>> would register fine
>> without the change of default algo.
>> * In X-Lite, use something similar to the following:
>> 	outbound proxy = pcscf.open-ims.test:4060
>> 	username = alice
>> 	domain = open-ims.test
>> 	authorization username = alice at open-ims.test
>>
>> Hope this helps,
>> -David
