Hi Arnold,

It depends on what type of authentication would you like to use. For
MD5, look at RFC 2617. For AKA look at RFC 3310.

The Authorization header must be present for AKA authorization. It does
work without, but the Authorization header is the only method of
informing the core what private identity (authentication identity) would
you like to use. Now we do have a fallback (which should actually be
used only for MD5 and Early-IMS and not AKA) that in case there is no
Authorization header present, then it extracts the private identity as
user at hostname from the public identity. This is why with the Alice or
Bob accounts it works. Because we have public identity
sip:alice at open-imst.test and private identity alice at open-ims.test. But
if private would be alice_impi at open-ims.test it won't work anymore.

So, if you can produce an Authorization header in every REGISTER, great.
If not, make sure that the private and public identities are the same,
except that the public can have extra scheme and parameters.

-Dragos
