Actually I just wanted to know how the priorization overall works, so in my 
configuration the Via- and Contact-headers do have the same port numbers.

>And now I try to answer you question. In case of NAT the network node
>detecting the NAT-situation (e.g. by comparing the IP address in UDP header
>with IP address in the topmost Via header field) adds a received-parameter
>(containing the IP-address where the request was received) to the Via 
>header
>field and also a rport-parameter (containing the port-number where the
>request was received). This network node is in our case the P-CSCF. Both
>parameters (received and rport) are then used to forward the response to 
>the
>correct NAT-ed address..

Thanks for the answer! And with the help of google, I found also another 
nice description about it with using rport as a search key
https://lists.cs.columbia.edu/pipermail/sip-implementors/2005-March/008560.html

The situation looks really such, that the UDP port number is added to the 
Via-header as a rport and therefore the Via-header port number is 
"overwritten". But I still have one concern: shouldn't the Contact-header in 
the end override all of the others? Or what is the reasoning for the header?
