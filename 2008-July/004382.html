Franz Edler wrote:
> Hi,
>
> I have now spent more time in analyzing the "qop" issue.
> There are two options for IMS-communicator to make it usable again with
> OpenIMSCore (r>=570):
> a) implement the qop algorithm
> b) do not implement the qop algorithm
>
> Option a) obviously requires more effort than b). It requires to dig deep
> into RFC2617 and correctly calculate the response (digest and checksum
> value).
>
> Option b) means to simply ignore the qop directive in the WWW-Authenticate
> header field of 401 Unautorized response. In this case the cnonce and the nc
> parameter value in the Authorization header field of the second REGISTER
> MUST not be included. This should automatically invoke the fallback solution
> for "qop not supported".
>
> Actually the IMS-communicator includes the cnonce and nc parameter value but
> does not correctly calculate the response (so its neither option a) nor
> option b)). So maybe it is simpler to not send the cnonce and nc parameter
> value and have the fallback to "qop not supported" method. This would make
> the IMS-communicator usable again with the actual release of OpenIMSCore.
>
> Maybe option b) is easier to implement.
