Actually it looks like those 2 come from some other client? the last
SUBSCRIBE, which has the Via contents the same as the data saved in the
Contact in the 200 OK for REGISTER, goes through normally.

Then for the call, the Via header is not the one that we can see in the
registration traffic. Either you used another client that did not
register or your client jumps ports and uses un-registered addresses.

Please send a trace showing both register and call, done with the same
client instance.

> Jeff,
>  
> I now have looked at the scenario (attached) and found:
> - The registration goes ok (200 OK)
> - But the PUBLISH and SUBSCRIBE requests after REGISTER are rejected with 
>   "403 Forbidden - Not Registered! You must register first with a S-CSCF"
>
> The function "P_is_registered()" checks if the originator contact of this
> message is registered at a S-CSCF. The check is based on the Contact header
> field. In both cases the result is negative because (my quick guess):
>
> - PUBLISH does not contain a Contact header field
> - SUBSCRIBE has a Contact header field but now uses TCP instead of UDP as
> with REGISTER.
