Hi,

 

Depends on the event you want to subscribe to. Seems like you need to
provision Public Service Identities (PSI) to your services (e.g.,
sip:as1 at example.com and sip:as2 at example.com). In case you haven’t done this
before it is almost the same as creating an IMPU, except you need to change
the “IMPU Type” to Distinct_PSI or Wildcarded_PSI (I think you need the
former in this scenario).

Hi all, 
 
We have 2 Application servers connected to the same Open IMS domain ( AS1
and AS2).
We would like to send a SUBSCRIBE from AS1 to AS2 through the CSCF and then
get a 200 OK answer back from AS2 to AS1 via CSCF.

When we send the message from AS1 to AS2 through the S-CSCF we get '404 Not
found - destination user not found on this S-CSCF'.
(We set the To-field to AS2 but we set a Route header to point to S-CSCF)
We have set up both App servers in the Open IMS on different ports and we
have given them own filer criterias that are built on the To-header.

Note: When we send SUBSCRIBE from a user to the servers it works perfect.
The message go througt to both of them depending on the To-header in the
message.
 
 
What could be wrong?
1. Should we send the message through the S-CSCF if we want to communicate
between 2 app servers in the same domain? Or should we go through I-CSCF?
2. Is there something that has to be set in some configuration file in the
system, so the servers can find eachother?
3. Is the  Profile Part Indicator of the IFC attached to the AS influencing
whether it should be registered or unregistered?
4. Each user profile has a service profile, which contains the respective
ifc/tp - how is the application server treated in this case, should the
application server have a service profile ?
