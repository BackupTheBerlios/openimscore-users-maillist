Hi Soshant,

The code for calling the P_is_integrity_protected() is commented - look
at pcscf.cfg:145:152.

You are right about what should be checked in the
P_is_integrity_protected() function. This is more of a stub right now
and requires some work, like all the IPSec stuff does, to build-up a
secure solution. One thing that certainly is missing is an IPSec SA
manager between our P-CSCF and the kernel implementation of IPSec. What
is in there now is just for prototype trialling and certainly not for a
deployment in an hostile environment.

Well, in any case, much could be improved with little efforts. Keeping
in mind that these changes would probably break the usage of non-3GPP
clients or NATed ones, I would be in favor of having 2 separate
pcscf.cfg - one for the strict case and one for the relaxed one. So if
you do any changes, we would greatly appreciate your contributions. If
those changes could be integrated without upseting other people - like
having some flags to control the whole operations, it would be even better.

-Dragos

Soshant Bali wrote:
> Hi Dragos,

> Is P_is_integrity_protected() called when an INVITE is received from
> the UE? If this function is not called then this can be a security
> vulnerability. Let there be two UEs (UE1 and UE2), and UE1 registers
> successfully. UE 2 is a malicious UE and is not registered but knows
> UE1s Contact info. Then UE2 can send an invite to pcscf, with Via
> header that is exactly the same as Via header coming from UE1. Since,
> P_is_integrity_protected is not called when INVITE is received, pcscf
> will not check whether or not the INVITE arrived protected. Also,
> since pcscf only compares Via to see whether UE is registered or not,
> and since via of UE2 INVITE is same as that of UE1, the INVITE will
> pass through pcscf.
>
> Also, the correct way to make sure that any message arrived protected
> is to check the (Source IP, Source Port, Destination IP and
> Destination Port) and see if these four match the security association
> that was created for this UE. The code for P_is_integrity_protected is
> only matching the port number in the Via to the port number in the
> created security association. In this case, it is very easy for a
> malicious UE to make the pcscf believe that the message arrived
> integrity protected, since all the malicious UE has to do is to form
> the Via header correctly.
>
> Thanks,
> Soshant
