Hmmmm... my point of view would be that the session case for
registration is only a mobile originating operation and never a
terminating one as the operation ends on the originating side. This is
because you can't really define a terminating side. Then, if you think
about how it is routed, you can argue that actually the I-CSCF after the
UAR/UAA "terminates" the operation towards the S-CSCF :)... Well, for
generality, I think that the session case should be ignored when IFC
checking for the 3rd party registration... but then you will need to
always specify a Method!=REGISTER if you want to catch all terminating
requests... So I will keep it as it is, with originating only. (or how I
hope that it is ;-) ).

-Dragos
