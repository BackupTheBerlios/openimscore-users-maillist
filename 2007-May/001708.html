> According to 3gpp specification:
>
>
> The Service-Route header is only applicable to the 200 OK of a
> register request. The UE will save the values aligned in this header
> to build Route header in subsequent requests.
> [This list saved in P-CSCF is used to validate routeing information
> built from initial request]
>
> thus
>
> Route /Service-Route header
>
> The Route header is built (mandatory) in subsequent requests (INVITE,
> SUBSCRIBE, MESSAGE, PUBLISH, etc) using the P-CSCF IP address or FQDN
> and the protected server port, and the values in recorded
> Service-Route header. The value of this header is compared with the
> user's associated Service-Route list in the P-CSCF to validate
> routeing information.
>
> else
>
> *P-CSCF returns a 400 **Bad request –not** **following indicated
> service routes*
>
>
> *I'm having the same problem with IMS Communicator. As I can
> understand from the traces, the client makes a presence event
> subscribe (for IM) during registration thus receiving a 403 forbidden.
> The thing is that the client makes the subscribe without waiting for
> the 200 OK thus not updating the service-route header. And afterwards
> subsequent presence subscription does not contain any route header and
> receiving 400 error messages.*
>
> *I think that it is a bug of IMS Communicator may be resolved in
> newest version.*
