Hello Ketzal,

are you sure that the terminating user was "not registered" at the
moment? From the LIA it seems that it was as there is an S-CSCF name
attached to it and this should only happen while the user is actually
registered or unregistered.

ketzal devims wrote:
> Hi everybody,
>
> I have a problem in a special case:
>
> I would like to test a service in the open source IMS.
> This service is a voice mail provided by mobicents.
>
> To execute this one, I need the following environment:
> - The terminating user must be unregistered
> - The HSS must have a trigger connected with the terminating user.
> This trigger is up for the 'INVITE' SIP Method, and the description
> case 'UNREGISTER'
> I have checked this trigger, It seems to be OK
>
> Then, when I tried to make a call between to users, the SIP request
> arrives to the I-CSCF, which must send a LIR Diameter request to know
> which S-CSCF is assigned to the terminating user.
>
> In the 3GPP TS29.229 Release 7, it is said that there is a Diameter
> answer for this exact case :
> "DIAMETER_UNREGISTERED_SERVICE (2003) :
> The HSS informs the I-CSCF that:
> - The public Identity is not registered but has services related to
> unregistered state
> - A S-CSCF shall be assigned to the user"
>
> So, I normally have to receive a 2003 in the LIA between the I-CSCF
> and the HSS, but I receive a "2001 - DIAMETER_SUCCESS" !!!
>
> A trace is attached for a better comprehension
>
