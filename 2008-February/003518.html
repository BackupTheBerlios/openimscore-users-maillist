 The OP/AMF values in the db are stored in tinyblob. The K is not, but
 now see it's varchar(255). It should be tinyblob. I am doing the changes
 right now...

 A dual representation would be messy in the web form there, so I will
 try something else: you can input/output it there for AKA in hex, just
 like the OP field, but it has to have exactly 16 bytes in binary! Else
 the input/output is treated as ASCII.

 BTW: the K in AKA is truncated at 16 bytes or \0 padded to the right up
 to 16 bytes.

 I think Jean-Yves was looking for this:

 update impi set op=x'6161616161....' where id=123;

 I couldn't find how to escape an arbitrary character inside a string in
 a query, but just some special ones and the rest seem to be written by
 mysqldump as non-ascii. But you can specify for blobs instead of a
 string a byte array in hex:
 http://dev.mysql.com/doc/refman/5.0/en/hexadecimal-values.html . But for
 varchar it does not work. Probably writting non-ascii chars would work,
 with the explicitly escaped exceptions from here:
 http://dev.mysql.com/doc/refman/5.0/en/string-syntax.html . Anyway, we
 should change that to tinyblob I would say.

 -Dragos
