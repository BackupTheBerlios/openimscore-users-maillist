Mikhanov Sergey wrote:
>
>    Hello,
>

>
> I found an issue related to orig/term dialogs handling in OIMS. It is
> a very possible situation that both of the parties involved in the
> call are located on the same node (same IP address), using same port
> and transport for communications. It is possible for SIP stack to
> unambiguously recognize which message belongs to which dialog (using
> combination of Call-ID, local and remote tag, as specified in RFC 3261).
>
>  
>
> IMS architecture assumes that initial INVITE from A to B should pass
> the following route (provided that no services are triggered, and both
> of the users are located in the same network, which is home network
> for them):
>
>  
>
> A -> P-CSCF -> S-CSCF ->* P-CFCS -> B
>
>  
>
> If A and B are on the same node and listening to the same port, i.e.
> sharing one stack instance, then when P-CFCS receiving INVITE marked
> with * it tries to fetch the saved dialog, using combination of
> Call-ID, host, port and transport. This dialog exists for this
> situation (while it shouldn’t be), see P_is_in_dialog function, and
> this INVITE is treated as Term_Subsequent response, while it should be
> Term_Initial.
>
>  
>
> My question is: why dialogs in OIMS are identified by Call-ID, host,
> port and transport instead of Call-ID, and tags as stated in RFC?
> Could this be an error?
>
>
